# AI Agent Development Blueprint: `jref config`

## Objective
Create a centralized, persistent environment configuration manager that acts as the single source of truth for runtime overrides, preferences, and environment variables across the `jref` CLI.

## Architectural Context & Constraints
* **Execution Pipeline:** Configuration loading must be synchronous and occur at the absolute start of the `main()` function execution thread in `src/index.ts`. 
* **State Merging:** The loaded configuration object must dynamically deep-merge into the runtime `CLIOptions` object, ensuring user preferences automatically propagate to all command contexts.
* **Target Environments:** Assume Linux/ARM pathing schemas. 

## Implementation Steps
1. **Schema Definition (`src/types/index.ts`):**
   * Define a comprehensive `zod` schema encompassing all valid configuration keys (e.g., default output style, global alias toggle, UI preferences, default `$JREF_BIN_PATH`).
2. **State Manager (`src/utils/config.ts`):**
   * Implement logic to read, parse, and merge configs from the local project `.jref/config.json` and the global `~/.jref/config.json`.
   * Create getter/setter utility functions to mutate the configuration on disk.
3. **Command Implementation (`src/commands/config.ts`):**
   * Build the CLI command to get/set individual values (e.g., `jref config set default_output json`).
4. **Interactive UI (`src/components/ConfigUI.tsx`):**
   * If the `--ui` flag is passed, spawn an `ink`-based rendering tree providing a visual dashboard to toggle local and global configuration settings. 

## Debugging Protocol
* If a configuration file exists but contains invalid JSON or fails schema validation, the CLI must NOT crash fatally. It should fallback to default hardcoded options, proceed with execution, and append a silent warning containing the Zod validation errors to `.jref/debug.log`. Console debugging workflows are strictly prohibited.
